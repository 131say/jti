# ADR: CadQuery Generation & Export Pipeline (v1.0)

## 1. Контекст и проблема

Воркер принимает `BlueprintPayload`, парсит геометрическую часть и генерирует 3D-модели через `cadquery`. Нужно зафиксировать минимальный набор примитивов и операций для v1.0 и надёжный экспорт для веба (Three.js) и симулятора (MuJoCo).

## 2. Поддерживаемые примитивы (Primitives v1.0)

Для снижения ошибок позиционирования в первой версии поддерживаются только два базовых тела:

- **`cylinder`**: параметры `radius` и `height`. Базовое построение: `cq.Workplane("XY").circle(radius).extrude(height)`.
- **`box`**: параметры `length`, `width`, `height`. Базовое построение: `cq.Workplane("XY").box(length, width, height)`.

**Решение:** типы `sphere` и `custom_profile` в v1.0 на стороне воркера отклоняются с явной ошибкой валидации.

## 3. Поддерживаемые операции (Operations v1.0)

Набор булевых операций ограничен отверстиями типа **`hole`**:

- Если `depth` — `"through_all"`, используется `cutThruAll()` (сквозное сверление).
- Если `depth` — положительное число — `cutBlind(depth)`.
- Центр плоскости входа — `position`, ось сверла — нормализованный вектор `direction`.

Реализация: `services/worker/generator.py` (`Plane` + `Workplane` + `circle` + cut).

## 4. Стратегия экспорта (Export Pipeline)

Сборки CadQuery (`cq.Assembly`) в MVP не используем. Генерируются **отдельные тела по `parts`**, затем они **объединяются** (`union`) в одно тело для экспорта (детали в одном начале координат; относительные трансформы между деталями в схеме v1 не задаются).

- **Симуляция / производство:** `.step` — `cq.exporters.export(model, "model.step")`.
- **Веб (Three.js):** **`model.stl`** — `cq.exporters.export(..., exportType="STL")`. Текущий стек CadQuery не регистрирует экспорт по расширению `.glb` без явного типа; glTF бинарный из коробки не используем. На фронте — `STLLoader` + `meshStandardMaterial` (см. `ModelViewer.tsx`).

## 5. Архитектура парсера (Worker Logic)

Модуль `services/worker/generator.py`:

1. Разбор `payload` → список тел по `geometry.parts`.
2. Для каждой части — построение solid по `base_shape` и `parameters`.
3. Операции (в будущем) — к соответствующему телу.
4. Объединение тел в одно тело (`union`).
5. Экспорт во временную директорию, затем загрузка в MinIO (`model.step`, `model.glb`).

## 6. Последствия

- **Плюсы:** узкий MVP, предсказуемый парсер, без фасок/скруглений в v1.
- **Минусы:** нет сложных криволинейных деталей и иерархии сборки; несколько частей визуально могут пересекаться в нуле — приемлемо до появления трансформов в схеме.
